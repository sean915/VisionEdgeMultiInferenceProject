1. 시스템 개요

RtmAlgorithmDll은 IVS 시스템에서 사용하는 알고리즘 DLL로, 영상 기반 결함 검출을 수행합니다.

  • Trigger 모델: Tab, Horn 객체 검출 (YOLO 형식)
  • Defect 모델: Crop 영역 분류 (Abnormal / Normal / Empty)
  • 인프런스 백엔드: ONNX Runtime 또는 TensorRT
  • 호출 방식: C# P/Invoke를 통한 DLL 로드 및 콜백 기반 비동기 결과 수신

3. Input 명세 (Client → RtmAlgorithmDll)

CNS/IVS APP이 RtmAlgorithmDll에 전달하는 입력은 설정(AlgorithmConfig)과 프레임(FrameData) 두 가지입니다.

3.1 API 호출 순서

1. CreateAlgorithm(config)     → handle 반환
2. SetResultCallback(handle, cb)  → 콜백 등록
3. Initialize(handle)             → 모델 로드만 (워커 미시작)
4. Run(handle)                    → 워커 스레드 시작 (triggerLoop, defectLoop)
5. PushFrame(handle, frame)       → 매 프레임 호출 (반복, Run 상태에서만)
6. Stop(handle)                   → 워커 중지 (멈춤 시, 재시작 시 Run만 호출)
7. Destroy(handle)                → 종료 시

멈춤 후 재시작: Stop → Run (Initialize 재호출 불필요)

3.2 실제 코드 구조

struct AlgorithmConfig {
    const char* triggerModelPath = nullptr;
    const char* defectModelPath = nullptr;
    float confThreshold = 0.5f;
};

struct Frame / FrameData {
    uint64_t index = 0;
    uint8_t* data = nullptr;   // BGR raw pointer
    int width = 0;
    int height = 0;
    int64_t timestamp = 0;
};

4. Output 명세 (RtmAlgorithmDll → Client)

RtmAlgorithmDll은 ResultCallback을 통해 판정 결과, 시각화 이미지, 에러 정보를 비동기로 전달합니다.

4.1 실제 코드 구조

내부 C++ 구조 (DetectOutputDto) — 콜백 호출 전 내부에서 사용:

// C API 콜백: ResultCallbackFunc
typedef void (*ResultCallbackFunc)(
    uint64_t frameIndex,
    int resultCount,
    const ResultItemC* results,
    int errorCode,
    const char* errorMsg,
    const uint8_t* imgData,
    int imgW, int imgH,
    int imgStrideBytes,
    int imgCvType
);

// 내부 DTO → C 콜백 매핑
struct DetectOutputDto {
    cv::Mat visualized_img;                    // → imgData, imgW, imgH, imgStrideBytes, imgCvType
    uint32_t takt_time;                        // (C API에는 미전달)
    std::vector<ResultItem> ab_results;        // 이상(불량) 결과
    std::vector<ResultItem> q_results;         // 의심 결과
    std::vector<ResultItem> normal_results;    // 정상 결과
    int error_code;                           // → errorCode
    std::string error_msg;                     // → errorMsg
};

struct ResultItemC {
    int defect_type;   // 0:NONE, 1:NG, 2:OK
    float score;
    BoxC box;
};

struct BoxC {
    int x1, y1, x2, y2;  // 좌상단(x1,y1), 우하단(x2,y2) 픽셀 좌표
};

4.3 출력 시점별 상세

호출 시점 resultCount results errorCode errorMsg imgData 시각화 내용

4.4 내부 결과 → C 콜백 매핑

Defect 완료 시 ab_results + q_results + normal_results를 순서대로 합쳐 results 배열로 전달합니다. RtmAlgorithmDll은 분류 모델이 단일 결과만 반환하므로 resultCount는 항상 1입니다.

box 좌표 의미: Defect 완료 시 results[0].box는 분류에 사용된 ROI 영역(원본 프레임 기준)입니다. Tab bbox를 crop하여 분류했을 경우 Tab 영역의 픽셀 좌표입니다.

내부 소스 defect_type 의미
ab_results 1 (NG) 불량 (Abnormal)
q_results 0 (NONE) 의심 (Questionable) 또는 에러
normal_results 2 (OK) 정상 (Normal)

4.5 DefectTypeEnum / ErrorCodeEnum 참고

DefectTypeEnum (defect_type):

값 이름 의미
0 NONE 판정 불가 (Questionable, Error, 파싱 실패)
1 NG 불량 (Abnormal)
2 OK 정상 (Normal)

ErrorCodeEnum (errorCode):

값 이름 의미
0 NO_ERROR 정상
1 FILE_NOT_FOUND 파일 없음
2 MODEL_LOAD_FAILURE 모델 로드 실패
3 INFERENCE_FAILURE 인프런스 실패
4 INVALID_INPUT 잘못된 입력
99 UNKNOWN_ERROR 알 수 없는 오류

4.6 C# P/Invoke 사용 참고

주의사항:

  • imgData: 콜백이 반환되면 포인터가 무효화됩니다. 사용하려면 콜백 내에서 Marshal.Copy 또는 Buffer.MemoryCopy로 즉시 복사하세요.
  • results: resultCount > 0일 때 Marshal.PtrToStructure로 ResultItemC 배열을 복사하세요. 포인터는 콜백 반환 후 무효화됩니다.
  • errorMsg: IntPtr인 경우 Marshal.PtrToStringAnsi로 문자열 복사.
// C# 콜백 예시 (구조만)
void OnResult(ulong frameIndex, int resultCount, IntPtr results,
    int errorCode, IntPtr errorMsg,
    IntPtr imgData, int imgW, int imgH, int imgStrideBytes, int imgCvType)
{
    // 1. imgData 복사 (필요 시)
    if (imgData != IntPtr.Zero && imgW > 0 && imgH > 0) {
        var bytes = new byte[imgStrideBytes * imgH];
        Marshal.Copy(imgData, bytes, 0, bytes.Length);
    }
    // 2. results 복사
    for (int i = 0; i < resultCount; i++) {
        var item = Marshal.PtrToStructure<ResultItemC>(results + i * Marshal.SizeOf<ResultItemC>());
    }
}

6.1 triggerLoop

역할: 프레임에서 Tab/Horn 검출 후, Horn이 위로 회전하는 시점에서만 Defect 큐로 전달.

  • 입력: frame_queue_owned_
  • 처리: Letterbox → Trigger 모델 → NMS → Tab/Horn bbox → Horn 방향 추적 (Up 시 캡처)
  • 출력: should_trigger 시 defect_queue_에 DefectJob push, 아니면 콜백(vis, 0)

6.2 defectLoop

역할: Tab ROI crop을 Defect 모델로 분류하여 Abnormal/Normal/Questionable 판정.

  • 입력: defect_queue_ (DefectJob: frame_bgr, crop_bgr, trig.tab/horn 등)
  • 처리: Crop → Letterbox → Defect 모델 → p_ab, p_no 파싱 → MIN_VALID(0.3), SURE(0.7) 기준 판정
  • 출력: 콜백(vis, results) — ab_results / q_results / normal_results

8. 참고사항

배포 및 운영 시 참고할 경로, 트리거 조건, 판정 기준입니다.

8.1 파일 경로

모델 메타 JSON은 D:/workspace/.../cuttermagazine_model 기준으로 로드됩니다. 실제 배포 시 경로 수정 필요.

8.2 트리거 조건

  • Tab + Horn 둘 다 검출 (score ≥ 0.4)
  • Horn 중심 Y가 이전 대비 위로 이동 (Up 방향) 시에만 Defect 큐로 전달
  • tracking_max_age=10 프레임마다 추적 상태 리셋

8.3 판정 기준

topScore 범위 최종 판정
< 0.3 (MIN_VALID) Error
0.3 ~ 0.7 Questionable
≥ 0.7 (SURE) && top==AB Abnormal (NG)
≥ 0.7 && top==NO Normal (OK)

이어서 RTM 전달용 정의서와 상황별 정리를 다룹니다.

10. [CNS 관점] IVS APP 연동 및 의사결정

CNS/IVS APP에서 RtmAlgorithmDll을 연동할 때의 처리 정책, 후처리, Final Decision 형식을 정리합니다.

10.1 비런타임/런타임 처리 의사결정 (LGES)

상황 Algo DLL 처리 기본 기능(RTSP) 처리 권장 (RtmAlgorithmDll 기준)
1) [비런타임] 사용자 멈춤 or config 변경 시 멈춤: Stop(handle). 재시작: Run(handle). config 변경 시: Destroy → CreateAlgorithm → Initialize → Run RTSP 수신 중단/재연결 멈춤 시 Stop, PushFrame 중단. 재시작 시 Run만 호출 (Initialize 불필요)
2) [런타임] PLC 설비 상태 on → off DLL은 PLC 미수신 — PushFrame 계속 가능 CNS가 RTSP/프레임 전달 중단 여부 결정 CNS에서 설비 off 시 PushFrame 중단 또는 빈 프레임 전달 정책 필요
3) [런타임] PLC 설비 상태 off → on PushFrame 재개 RTSP 재연결 CNS에서 설비 on 시 PushFrame 재개

참고 (IVS PKG Prewelding): IVS 참고 구조에서는 PLC Task가 Cell ID, Pallete, Busy를 읽고 설비 상태에 따라 동작을 제어합니다. RtmAlgorithmDll은 PLC 입력을 받지 않으므로, CNS/IVS APP이 PLC 상태를 읽어 PushFrame 호출 여부를 제어해야 합니다.

10.2 IVS APP 후처리 로직 공유

Algo DLL은 판정 결과(defect_type, score, box)만 반환합니다. 후처리(알람 매핑, CSV/로그 저장, PLC 알람 전송 등)는 CNS/IVS APP에서 수행합니다.

후처리 항목 담당 비고
defect_type → 알람 코드 변환 CNS NG=1, OK=0 등 PLC 규격에 맞게 매핑
이미지/CSV/로그 저장 CNS ResultCallback의 imgData 복사 후 저장
PLC 알람 출력 CNS DLL은 PLC 쓰기 없음

10.3 Final Decision 값 — 숫자 vs 문자열

형식 현재 DLL 출력 권장
숫자 defect_type: 0, 1, 2 (int) PLC/시스템 연동 시 숫자 사용 권장. 매핑 간단
문자열 errorMsg 등에 "Abnormal", "Normal" 텍스트 포함 가능 로그/UI 표시용으로 CNS에서 defect_type → "NG"/"OK"/"NONE" 변환

현재: DLL은 defect_type (int)로 전달. CNS에서 필요 시 문자열로 변환.
